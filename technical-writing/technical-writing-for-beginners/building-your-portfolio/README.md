# Building Your Technical Writing Portfolio

_Showcase your thinking, not just your output._

***

## 🪜 Purpose of This Chapter

To help beginners:

* Understand what a strong technical writing portfolio includes (and excludes)
* Structure GitHub repos that demonstrate real-world documentation thinking
* Use projects — even self-initiated ones — as proof of clarity, not code
* Build visibility and trust as a writer, not just a “content creator”

This chapter shows you how to **think like a documentation designer**, not just a doc writer — and then showcase it effectively.

***

## 🎯 Why a Portfolio Even Matters

You don’t need experience in a big tech company to prove your skill.

> You need _proof that you can help someone understand something they didn’t before._

Your portfolio is your credibility. It’s your working demo.\
And it’s especially important when you're:

* Transitioning from another field
* Self-taught
* Working with open-source projects
* Building your presence as a technical communicator

***

## 🧱 What Makes a Strong Portfolio?

A good portfolio doesn’t just say:

> “I wrote docs for a product.”

It shows:

> “Here’s how I structured information, clarified complexity, and made it easier to use.”

📌 **Tip:** Include _process_, not just finished docs.

***

## 📂 Suggested Structure (Simple GitHub Setup)

```bash
technical-writing-portfolio/
├── README.md                     # Explains who you are + how this portfolio is structured
├── how-to-guides/
│   └── guide-example.md
├── rewritten-docs/
│   └── original-vs-rewrite.md
├── api-docs/
│   └── simple-api-reference.md
├── tutorials/
│   └── first-tutorial.md
├── reflections/
│   └── what-i-learned-from-this-project.md
```

✅ Each folder = a category of your work\
✅ Each file = a doc that demonstrates clarity, structure, and empathy\
✅ Bonus if you link to **live rendered versions** (e.g., MkDocs site, Notion, Confluence, GitHub Pages)

***

## 📘 What to Include

| Section            | Purpose                                                                           |
| ------------------ | --------------------------------------------------------------------------------- |
| `README.md`        | Explain the repo, who it’s for, and what you’re demonstrating                     |
| Use cases or goals | Who the doc is meant to help                                                      |
| Clear structure    | Subfolders, TOCs, and filenames that tell a story                                 |
| Realistic examples | Based on tools, systems, or workflows people know                                 |
| Commentary         | Explain decisions, rewrites, and challenges (in comments or a reflections folder) |

🧠 **Pro Tip:** Use comments inside your Markdown files to show thinking:

```markdown
<!-- I rewrote this section to clarify the goal for non-technical readers -->
```

✅ Add short intros at the top of each doc:

```markdown
> This how-to was written to help a first-time user set up Slack reminders using a non-technical UI.
```

## ✍️ Sample Project Ideas

You don’t need access to a product team.\
You just need curiosity and structure.

| Project Type                                              | What It Shows                       |
| --------------------------------------------------------- | ----------------------------------- |
| Rewrite an open-source README                             | Clarity, tone, onboarding skills    |
| Write a how-to for Notion, Trello, Canva, etc.            | Task-based writing for real tools   |
| Create your own imaginary product                         | UX-thinking, from scratch           |
| Document a simple API (e.g. OpenWeather, JSONPlaceholder) | Technical precision                 |
| Convert a YouTube tutorial into steps                     | Translation + instructional writing |
| Create comparison docs (before vs after)                  | Editing and improvement mindset     |
| Build a mock onboarding guide                             | User empathy and flow awareness     |

## 🛠 Tools to Use in Your Portfolio

| Tool                         | Purpose                                |
| ---------------------------- | -------------------------------------- |
| GitHub / GitLab              | Store and showcase docs (public repos) |
| MkDocs / Docusaurus          | Publish Markdown docs beautifully      |
| Obsidian / Typora / VS Code  | Local writing and preview              |
| CleanShot / Snagit           | Annotated screenshots if needed        |
| Grammarly / Hemingway / Vale | Language polish and linting            |
| Netlify / GitHub Pages       | Deploy live versions of your docs      |
| Notion (optional)            | Share research or concept maps         |

***

## 📋 What Recruiters or Hiring Managers Look For

| Trait               | How to Show It                           |
| ------------------- | ---------------------------------------- |
| Clarity             | Task-based how-to guides                 |
| Initiative          | Rewriting docs you didn’t “own”          |
| Organization        | Logical folder structure                 |
| Empathy             | Reader notes, FAQs, personas             |
| Attention to Detail | Consistent formatting, links, grammar    |
| Process Thinking    | Comments, reflection files, before/after |

***

## 📢 Bonus: Document the Docs

📘 Create a file like `portfolio-explained.md` where you:

* Walk through how you structured the portfolio
* Talk about what decisions you made and why
* Share what you learned by writing each doc

This **meta-layer** builds trust and shows you're not just writing — you're _thinking about how people learn._

***

## ✅ Portfolio Quality Checklist

* [ ] Demonstrate more than one doc type (e.g., how-to, tutorial, API)?
* [ ] Include at least one original and one rewritten doc?
* [ ] Include a README that explains purpose and navigation?
* [ ] Offer reflection on decisions made while writing?
* [ ] Use clear file/folder structure?
* [ ] Feel human, helpful, and usable?

***

## 🧭 Final Thought

> Your portfolio isn’t a gallery.\
> It’s a proof-of-understanding machine.

Make it:

* Navigable
* Purpose-driven
* Friendly to both readers _and_ recruiters

Show not just _what_ you wrote — but _why_ it works.
