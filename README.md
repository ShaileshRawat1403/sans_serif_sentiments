## Overview (Purpose)

> Writing isn’t just about inspiring it's more about informing, It’s not enough to tell users _what to do_—you have to show them _how_ and make sure they know _why they’re doing it_.

***

### Problem

Too many user guides:

* Start with steps before establishing relevance or context
* Forget that users aren’t all the same—some are writers, some developers, some just lost
* Treat clarity like an optional feature instead of a core function

As a result, documentation becomes a maze of instructions—no entry point, no map, no motivation.

***

### Rule

**Every guide should begin with an intent** that:

1. **Identifies** the role(s) the document serves (writer, editor, dev, etc.)
2. **Defines** the exact problem it helps solve
3. **Guides** the reader on how to use the doc most effectively

This creates orientation, sets expectations, and invites the reader in.

***

### Example

❌ **Wrong** (too eager, no context):

```markdown
# How to Contribute
Fork the repo, clone it, create a new branch...
```

✅ **Right** (context-first clarity):

```markdown
# Purpose of This Guide

This document helps **first-time contributors** understand how to work with the `sans-serif-sentiments` repository.  
You’ll learn **how the repo is structured**, **what the style rules are**, and **how to contribute confidently** without breaking anything.

# How to Contribute
1. Fork the repo…
2. Clone it locally…
```

Marketing was oK but it came with its usual _kit-kit_.\
So, as a writer, I caught the nearest rhyme—and found **Git to commit**.

What started as a pun turned into a revelation.

Under the rock called “Marketing” lies a world of Git—a place where writing isn’t dressed up, it’s tracked.\
Where edits aren’t hidden in redlines but logged in real time.\
Where you don’t pitch, you version.\
And you don’t persuade—you prove.

Here’s how the shift feels:

|                | Marketing                          | Git                                     |
| -------------- | ---------------------------------- | --------------------------------------- |
| **Truth**      | Bend it to fit a story             | Brutally honest (tracks every mistake)  |
| **Versioning** | FINAL\_final\_v2\_REAL.pptx        | `git commit -m "actual change"`         |
| **Audience**   | Persuade                           | Inform                                  |
| **Mistakes**   | Smooth over                        | Preserve and reflect                    |
| **Identity**   | Copywriter, Content Designer, etc. | Technical Writer, Information Developer |
| **Clarity**    | Optional                           | Mandatory                               |
| **Results**    | Vibes-based                        | Version-controlled impact               |
